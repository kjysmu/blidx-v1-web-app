# Deploying Blidx to Render staging

The repo ships a `render.yaml` blueprint that deploys the FastAPI app, serves
the web app from `/`, and mounts a 1 GB persistent disk at `data/` so demo
state survives restarts.

## One-time setup

Production and staging deployments refuse to start with unsafe authentication or
LinkedIn token settings. Configure these values before deploying:

- `ENVIRONMENT=production`
- `DEBUG=false`
- `USE_DATABASE_STORAGE=true`
- `DATABASE_URL` pointing to the Render Postgres database
- `JWT_SECRET_KEY` set to a unique random value of at least 32 characters
- `LINKEDIN_TOKEN_ENCRYPTION_KEY` set to a separate unique random value of at
  least 32 characters

Changing `JWT_SECRET_KEY` invalidates existing Blidx login tokens. Changing
`LINKEDIN_TOKEN_ENCRYPTION_KEY` requires users with an existing LinkedIn
connection to reconnect.

   `https://blidx-v1-web-app.onrender.com/auth/linkedin/callback`.

short-lived, one-time, and tied to the signed-in Blidx account. Text publishing
uses LinkedIn's versioned `/rest/posts` API with `w_member_social`.

## Every deploy

1. **Run the required local gate before pushing:**

   ```bash
   python scripts/release_gate.py
   ```

2. Push to `main`. GitHub Actions reruns the same gate in a clean Linux
   environment. The Blueprint sets `autoDeployTrigger: checksPass`, so Render
   waits for that CI check instead of deploying an unverified commit.
3. Render builds with
   `pip install -r requirements.txt` and starts
   `uvicorn app.main:app --host 0.0.0.0 --port $PORT`.
4. Wait for the health check to go green.
5. **Run the smoke test** against the live URL:

   ```bash
   python scripts/smoke_staging.py https://<service>.onrender.com
   ```

   It registers a throwaway user (never touches the shared public demo) and
   walks health â†’ integrations â†’ onboarding â†’ chat â†’ draft â†’ approve. Exit
   code 0 means ship it. It also tells you explicitly whether drafts came
   from Claude or the template fallback â€” the main thing to confirm on
   staging.

## Known staging caveats

- Auth users, profiles, Content Bank entries, drafts, and chat messages use
  Render Postgres when `USE_DATABASE_STORAGE=true`. Existing JSONB workspace
  records are imported into the relational workspace tables on first access.
- File-backed JSON storage remains available only for local development with
  `USE_DATABASE_STORAGE=false`.
- `POST /api/reset` clears the shared public demo; per-user workspaces are
  untouched.
- The `/admin` route is HTTP Basic-only â€” keep the credentials in a password
  manager, not in chat or docs.
