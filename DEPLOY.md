# Deploying Blidx to Render staging

The repo ships a `render.yaml` blueprint that deploys the FastAPI app, serves
the web app from `/`, and mounts a 1 GB persistent disk at `data/` so demo
state survives restarts.

## One-time setup

1. **Create the service** — in Render: New → Blueprint → point at this repo.
   The blueprint provisions `blidx-v1-web-app` (Python 3.12, starter plan,
   health check on `/health`).
2. **Set the secret env vars** (Render → Environment). The blueprint declares
   them with `sync: false`, so the deploy will not go live until they exist:

   | Variable | Purpose | Without it |
   |---|---|---|
   | `ANTHROPIC_API_KEY` | Claude-powered Mira (chat, angles, drafts, revisions) | Deterministic fallback generator |
   | `LINKEDIN_CLIENT_ID` / `LINKEDIN_CLIENT_SECRET` | LinkedIn OAuth publishing | Manual copy & open fallback |
   | `LINKEDIN_REDIRECT_URI` | OAuth callback, must be registered in the LinkedIn app | OAuth connect fails |
   | `LINKEDIN_TOKEN_ENCRYPTION_KEY` | Encrypts each user's LinkedIn token at rest | Falls back to the JWT secret |
   | `ADMIN_USERNAME` / `ADMIN_PASSWORD` | HTTP Basic auth for `/admin` | `/admin` disabled |

   `JWT_SECRET_KEY` is auto-generated by the blueprint. `DATABASE_URL` must point
   to the Render Postgres database; the blueprint enables relational workspace
   storage with `USE_DATABASE_STORAGE=true`.
3. **LinkedIn app config** — add this exact staging callback to the LinkedIn
   app's authorized redirect URLs:
   `https://blidx-v1-web-app.onrender.com/auth/linkedin/callback`.
   `app.blidx.com` should only replace it after DNS routes that hostname to the
   service. Until one exact callback matches, the manual fallback still works.

LinkedIn uses `/auth/linkedin/callback` as the only callback. OAuth state is
short-lived, one-time, and tied to the signed-in Blidx account. Text publishing
uses LinkedIn's versioned `/rest/posts` API with `w_member_social`.

## Every deploy

1. Push to `main` (or click Manual Deploy). Render builds with
   `pip install -r requirements.txt` and starts
   `uvicorn app.main:app --host 0.0.0.0 --port $PORT`.
2. Wait for the health check to go green.
3. **Run the smoke test** against the live URL:

   ```bash
   python scripts/smoke_staging.py https://<service>.onrender.com
   ```

   It registers a throwaway user (never touches the shared public demo) and
   walks health → integrations → onboarding → chat → draft → approve. Exit
   code 0 means ship it. It also tells you explicitly whether drafts came
   from Claude or the template fallback — the main thing to confirm on
   staging.

## Known staging caveats

- Auth users, profiles, Content Bank entries, drafts, and chat messages use
  Render Postgres when `USE_DATABASE_STORAGE=true`. Existing JSONB workspace
  records are imported into the relational workspace tables on first access.
- File-backed JSON storage remains available only for local development with
  `USE_DATABASE_STORAGE=false`.
- `POST /api/reset` clears the shared public demo; per-user workspaces are
  untouched.
- The `/admin` route is HTTP Basic-only — keep the credentials in a password
  manager, not in chat or docs.
